PRD — Automated TAT & Batch Monitoring System (Node.js + BullMQ Edition)

Version: 3.0 (Stack-Specific)
Stack: Node.js (Express) + BullMQ (Redis) + MongoDB Atlas
Architecture Style: Event-Driven Microservice (Modular Monolith)

1. System Goal

Build a fully automated, API-driven lab tracking system that:
- Receives sample data via SAP Webhooks
- Validates and normalizes multi-test payloads
- Calculates dynamic ETA based on EDOS batch windows (Asia/Kolkata)
- Predicts SLA breaches and calculates overage minutes
- Handles "Missed Batch" scenarios as high-priority events
- Fires real-time alerts (Console, Email, MongoDB Persistence)
- Provides a comprehensive Dashboard API

2. Architecture Design

The system is split into two main logical processes (runable together or separately).

Backend: node-service/
│
├── entry-regulator/   → Express Application
│   ├── POST /webhook  → Validates incoming SAP payload
│   └── BullMQ/        → Pushes jobs to Redis (sample-processing queue)
│
├── processing-engine/ → BullMQ Worker (Node runtime)
│   ├── EDOS Loader    → Parses CSV/JSON test rules
│   ├── Batch Assigner → Core logic (Batch, ETA, SLA, Breach)
│   └── Alert Service  → Console, Email, and DB Alerting
│
├── storage/           → MongoDB Atlas
│   ├── Samples        → Permanent record of all calculated TATs
│   └── Alerts         → Persistent log of all system alerts
│
└── dashboard-api/     → REST Endpoints
    ├── /api/samples   → List/Filter all samples
    ├── /api/stats     → Aggregate metrics
    ├── /api/batches   → Batch utilization view
    └── /api/alerts    → Recent alerts + summary

3. Database Design (MongoDB Optimized)

Collection: samples
{
  _id: ObjectId,
  sample_id: "SAP-12345",   // External ID from SAP
  test_id: "TID-001",
  test_name: "test_1",
  test_code: "ELISA023",
  received_at: ISODate,     // Injected by SAP or Regulator
  agreed_tat_hours: Number, // SLA requirement
  
  batch_id: String,         // Computed: BATCH-{CODE}-{DATE}-{TIME}
  batch_run_start: ISODate,
  batch_cutoff: ISODate,
  
  eta: ISODate,             // Computed ETA
  sla_deadline: ISODate,    // Computed derived SLA
  breach_flag: Boolean,     // True if ETA > SLA
  overage_minutes: Number,  // Overage if breached
  
  missed_batch: Boolean,    // True if received after cutoff
  delay_reason: String,     // Human-readable reason for delay
  status: "assigned",       // pending, assigned, delayed, error
  processed: Boolean,
  
  createdAt, updatedAt
}

Collection: alerts
{
  _id: ObjectId,
  type: "MISSED_BATCH",     // MISSED_BATCH, SLA_BREACH, DELAY_ESCALATION
  sample_id, test_name,
  priority,
  alert_data: {},           // Full snapshot of timers at alert time
  acknowledged: Boolean,
  createdAt
}

4. Event-Driven Flow (NO Polling)

Solution: BullMQ + Redis
Entry Regulator receives sample → Enqueues job to "sample-processing"
Worker listens to Redis → Picks up job immediately
Result saved to MongoDB → Dashboard updates on next fetch

5. Core Modules

Module A — Webhook Entry Regulator (src/routes/webhook.js)
- Normalizes single or multi-test (`tests: []`) payloads.
- Splits multi-tests into individual jobs.
- Validates all fields: required (sample_id, test_name, received_at, tat_hours).

Module B — EDOS Loader (src/services/edosLoader.js)
- Loads 1070+ test rules from CSV (or fallback `data/edos.json`).
- Parses human-readable schedules ("Tue / Fri 6 pm").
- Parses complex TAT rules ("Next Day 8 pm", "3rd Day 5 pm").

Module C — Batch Assigner (src/services/batchAssigner.js)
- Core Logic: Converts ISO string → Asia/Kolkata (IST).
- 30-Day Window Scan: Finds first available batch where `cutoff_time >= received_time`.
- ETA Computation:
  - If Deadline Hour: `batch_run_start + days @ hour`
  - If Relative: `batch_run_start + tat_minutes`

Module D — Alert Service (src/services/alertService.js)
- Persistent: Saves all alerts to MongoDB.
- Visual: Colorized console boxes for terminal monitoring.
- Email: SMTP-based plain-text summary (Sample, Priority, ETA, Reason, Action).

Module E — Dashboard API (src/routes/...)
- Samples: Paginated, filterable listing.
- Stats: Real-time counts of Breached vs Assigned.
- Batches: Aggregated view of Sample Load per Batch.
- Alerts: Recent failures and "Missed Batch" escalation.

6. Concurrency and Resilience
- BullMQ Retries: Handles temporary failures (e.g. DB connection) with exponential backoff.
- Graceful Shutdown: Processes finish cleanly (server/worker) on SIGINT/SIGTERM.
- EDOS Fallback: System starts even if CSV is missing via JSON fallback.

7. Demo/Verification Flow
1. Submit POST /webhook
2. BullMQ job created immediately.
3. Worker processes:
   - "Missed Batch" alert triggers red console box if received after hours.
   - "SLA Breach" alert triggers yellow console box if ETA > SLA.
   - Sample saved/updated in MongoDB.
4. Dashboard API returns latest computation result instantly.

8. Architecture Insight
You are building an Automated Scheduling & Prediction Engine.
- Event-Driven (BullMQ)
- Timezone Aware (Asia/Kolkata)
- Resilient (Graceful Shutdown + EDOS Fallback)
- Fully Persistent (MongoDB Alerts + Samples)
